============================================================
1) PREPARER UN PIPELINE DE TRAITEMENT D'IMAGES MEDICALES
============================================================

1.1 Structure et indexation des donnees
- Dataset total : 5 856 images.
- Repartition des splits :
  - Train : 5 216 (89.07%)
  - Validation : 16 (0.27%)
  - Test : 624 (10.66%)
- Le notebook 01 construit des DataFrames indexes (path, label, class_name, split)
  pour train/val/test, a partir de data/chest_xray.

1.2 Qualite des donnees
- Verification d'integrite effectuee : aucun fichier corrompu detecte dans train/val/test.
- Coherence des chemins et des labels validee apres indexation.

1.3 Variabilite des images et implications
- Forte heterogeneite de resolution observee :
  - Train : 4 366 tailles uniques
  - Test : 598 tailles uniques
- Images majoritairement en niveaux de gris (avec quelques cas en 3 canaux).
- Implication : besoin d'un pretraitement standardise avant apprentissage.

1.4 Pipeline de pretraitement (Notebook 02)
- Transformations de base/evaluation :
  - Resize(256)
  - CenterCrop(224)
  - ToTensor()
  - Normalize(mean=ImageNet, std=ImageNet)
- Transformation train (augmentation) :
  - RandomRotation(5) ajoutee aux operations precedentes.
- Chargement PyTorch : Dataset personnalise + DataLoader.
- Parametres loader observes : batch_size=32, seed=42, pin_memory=True.

1.5 Gestion du desequilibre de classes
- Distribution des classes :
  - Train : PNEUMONIA 3 875 (74.29%) vs NORMAL 1 341 (25.71%)
  - Val : 8 / 8 (50/50, mais tres petit echantillon)
  - Test : PNEUMONIA 390 (62.5%) vs NORMAL 234 (37.5%)
- Le notebook prevoit la ponderation de la classe positive via pos_weight
  dans BCEWithLogitsLoss pour limiter le biais du desequilibre.


============================================================
2) CONCEVOIR ET ENTRAINER UN CNN POUR LA CLASSIFICATION BINAIRE
============================================================

2.1 Architecture du modele (BaselineCNN)
- Entree : images RGB 224x224.
- Bloc convolutionnel (3 niveaux) :
  - Conv(3->32) + ReLU + MaxPool
  - Conv(32->64) + ReLU + MaxPool
  - Conv(64->128) + ReLU + MaxPool
- Classifieur :
  - Flatten -> Linear -> ReLU -> Dropout(0.5) -> Linear(1 logit)
- Tache : classification binaire (NORMAL=0, PNEUMONIA=1).

2.2 Strategie d'entrainement
- Fonction de cout : BCEWithLogitsLoss (avec pos_weight si active).
- Optimiseur : Adam.
- Hyperparametres observes dans les cellules :
  - lr = 1e-3
  - weight_decay = 1e-4
  - entrainement effectif du run sauvegarde sur 20 epoques.
- Fonctions metier implementees : train_one_epoch, validate_one_epoch, fit.

2.3 Comportement du run (courbes train/val)
- train_loss : 0.2303 -> 0.0446 (forte baisse)
- train_acc : 0.9095 -> 0.9824 (forte hausse)
- val_loss : minimum ~0.4038 puis remontee (~0.7735 en fin de run)
- val_acc : fluctue entre 0.5625 et 0.8125 (fin ~0.6875)
- Lecture technique : surapprentissage (overfitting) apres les meilleures epoques de validation.


============================================================
3) EVALUER LE MODELE AVEC DES METRIQUES ADAPTEES
============================================================

3.1 Metriques finales sur le test (624 images)
- test_loss : 1.5617
- Accuracy : 0.7644 (76.44%)
- Precision : 0.7314
- Recall (sensibilite) : 0.9846
- F1-score : 0.8393
- ROC-AUC : 0.8576

3.2 Matrice de confusion (derivee du notebook 02)
- TP = 384
- TN = 93
- FP = 141
- FN = 6

3.3 Interpretation des resultats
- Le modele detecte tres bien la classe PNEUMONIA (recall tres eleve).
- Peu de faux negatifs (FN=6), ce qui est positif pour le depistage.
- Beaucoup de faux positifs (FP=141), ce qui reduit la specificite
  et augmente les alertes inutiles.
- Niveau de performance global correct pour un baseline CNN,
  mais insuffisant pour une decision clinique autonome.

3.4 Conclusion d'evaluation
- Le choix des metriques (Accuracy, Precision, Recall, F1, AUC + matrice de confusion)
  est adapte au probleme binaire medical.
- Le modele est exploitable comme base de depistage assiste,
  avec besoin d'ameliorations (seuil, calibration, architecture plus robuste,
  validation plus solide) avant usage plus avance.
